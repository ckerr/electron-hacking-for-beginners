#### What code is in Electron? Electron is Chromium + Node.js + native OS bits. #### Where does it go? It's all statically linked. ##### We build the whole world? Mosly, yes. XXX When we're building Electron, what do we build? So, everything Chromium needs, everything Node needs. If you run `ldd electron` on Linux to see what shared libraries, what DLLs, it's using, you get libffmpeg, GTK+ and its dependencies. And that's about it. Like Chromium, for the most part if we need it, we compile it in. You don't see dynamically linked skia (the 2d graphics library Chromium uses for rendering) You don't see media libraries like libwebm You don't see node or libuv or v8. It's all compiled together. And there's upside to that too, such as making sure chromium and node are built with compatible build flags

So now you have the tools, it's time to get the code.

Enable caching: ```sh $ export GIT_CACHE_PATH="${HOME}/.git_cache" $ mkdir -p "${GIT_CACHE_PATH}" ``` Get the code: ```sh $ mkdir electron-gn && cd electron-gn $ gclient config \ --name "src/electron" \ --unmanaged \ https://github.com/electron/electron $ gclient sync --with_branch_heads --with_tags # This will take a while, go get a coffee. ``` XXX The important thing here you want to make sure you have a cache enabled. I'll show you why in a moment, but the way you enable it is by creating a cache directory and setting the environment variable `GIT_CACHE_PATH` to it. Once you have that, you run `gclient config` for first-time setup and `gclient sync` any time after that when you want to refresh your files. It's like a meta pull + submodule update.
**GClient Cost (Time)** First-time run (no cache): ```sh $ TIME="%e seconds" /usr/bin/time gclient sync \ --with_branch_heads --with_tags --delete_unversioned_trees ... 1218.00 seconds # 20 minutes, 18 seconds ``` With a populated cache: ```sh $ TIME="%e seconds" /usr/bin/time gclient sync \ --with_branch_heads --with_tags --delete_unversioned_trees ... 250.00 seconds # 4 minutes, 10 seconds ``` XXX No huge surprise here: the first run is quite expensive. That 20 min was with a good internet connection. but once you have the cache, resyncing is pretty cheap. Most of that four minutes is spent in repo hooks, for example re-applying Electron's Chromium patches. And how big is that cache?
**GClient Cost (Space)** ```sh $ du -sh ~/.electron-cache/git-cache 16G ``` XXX So when I said "imagine how many repos you have to clone" earlier.. There's your answer: 16 GB worth. Let's take a really quick look at the source.
### Electron's source code 1. `spec/` is the tests 1. `script/` is where all the extra tools go, e.g. release notes generator 1. `lib/` the public-facing Javascript 1. `atom/` is the C++ -- e.g. the Chromium subclasses, delegates and observers. XXX In a lot of ways Electron is just another Node app. You know what a `spec/` directory does. * `lib/` is the javascript. If we can implement a feature in JS, it goes here. But much of it is glue between our public API and the C++ backend. Atom is the next layer, the C++ code between lib and Chromium. Most of it exists to expose pieces of Chromium to Electron app devs. For example the largest class in Electron is atom_api_web_contents. Let's take at its header file to see what it's about:
** Excerpt from bhe biggest, scariest class in Electron (atom::api::WebContents)** ```cpp void LoadURL(const GURL& url, const mate::Dictionary& opts); base::string16 GetTitle() const; bool IsLoading() const; void Stop(); void ReloadIgnoringCache(); void GoBack(); void GoForward(); bool IsCrashed() const; void SetUserAgent(const std::string& user_agent); std::string GetUserAgent(); void InsertCSS(const std::string& css); void OpenDevTools(mate::Arguments* args); ``` XXX Not completely unexpected, right? This is the largest class we've got. And you understand a lot of the ideas here already. To be clear: no, I'm not saying Electron is easy. Some of it is, but a lot of it isn't. But if you've written Electron apps, if you've used `win.loadURL()` or `win.getTitle()`, you aleady see the big picture, and that can be your guide when you're reading and learning the littler pictures. The implementation may be unfamliliar, but `OpenDevTools()` isn't alien. Right?
